# AIエージェントで「報告の手間ゼロ」のデイリーレポートを実現した話

## はじめに

「メンバーの進捗、ちゃんと把握できていますか？」

開発チームのPM（プロダクトオーナー）にとって、この問いに自信を持って「はい」と答えるのは意外と難しいものです。特に、メンバーの労働時間帯が異なり、毎朝のスタンドアップミーティングを全員で行えないチームでは。

私たちのチームは6名のエンジニアで構成されていますが、各メンバーの稼働時間が異なり、**全員が同時にオンラインになる時間帯がほとんどありません**。デイリーの朝会は物理的に設けられない。かといって、「Slackで毎朝報告してください」と求めるのも、エンジニアの貴重な開発時間を「報告」という雑務に費やすことになります。

この記事では、**AIエージェント（LLM + MCP）を活用して、メンバーに追加の報告作業を一切課さずに、既存の開発ツールのデータだけでデイリーレポートを自動生成する仕組み**を紹介します。

## 背景・課題

### PM側の悩み：「把握したいが、聞くのも申し訳ない」

PMとして、各メンバーの状況を詳細に把握したい要望は当然あります。

- いまどのタスクに取り組んでいるのか
- マイルストーンに対してオンスケか遅れ気味か
- PRはレビュー待ちで詰まっていないか
- 何か困っていることはないか

しかし現実には、Backlogの課題一覧を開いて一つずつ確認し、GitHubのPR状況を巡回し、メンバーの日報を読み…という作業を毎朝やるのは、PM側もかなりの時間を取られます。

### メンバー側の悩み：「報連相は大事だけど、時間を取られる」

一方のメンバー側にも葛藤があります。報連相の重要性は理解している。でも、「今日の作業予定」「昨日の進捗」をフォーマットに沿って毎朝書くのは、正直なところ**優先度が低い**。目の前のコードを書くことに集中したい。

結果として、「報告してほしい」PMと「報告する暇があったらコードを書きたい」エンジニアの間で、**お互いに遠慮しながら中途半端な状態**が続いていました。

### 本質的な問い：「報告」に本当に新しい作業が必要なのか？

ここで立ち止まって考えました。

メンバーは日々、すでに多くの「痕跡」を残しています。

- **Backlog** にタスクのステータスを更新している
- **Stock** に日報メッセージを書いている（個人のメモレベルで）
- **GitHub** にコミットし、PRを出している

これらの既存データを**集約・分析するだけ**で、「いま誰が何をしていて、どのくらい進んでいるか」は相当の精度で把握できるはずです。

**メンバーに新しい作業をさせるのではなく、既にやっていることから情報を吸い上げる** — これがコンセプトでした。

## アプローチ

### アーキテクチャ

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Backlog    │     │    Stock     │     │   GitHub    │
│  (課題管理)   │     │  (日報メモ)   │     │  (PR/コード) │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                    │
       │    REST API       │    DB (読取専用)     │    gh CLI
       │                   │                    │
       ▼                   ▼                    ▼
┌──────────────────────────────────────────────────────┐
│              AIエージェント（LLM + MCP）                │
│                                                      │
│  1. データ収集     →  3データソースから自動取得          │
│  2. 進捗分析       →  マイルストーン突合・PR状態判定     │
│  3. レポート生成   →  テンプレートに沿って統合出力       │
└──────────────────────────────────────────────────────┘
       │
       ▼
  デイリーレポート（Markdown）
```

ポイントは、**人間が新たに入力する情報がゼロ**であること。全て既存のツールに蓄積されたデータだけで構成されています。

### 3つのデータソースと取得する情報

#### 1. Backlog（課題管理）— 「何をやるべきか」の計画データ

Backlog API から、各メンバーの進行中課題を取得します。

```bash
# メンバー別の進行中課題を一括取得
curl -s "https://{space}.backlog.com/api/v2/issues?apiKey=$API_KEY \
  &projectId[]={project_id} \
  &statusId[]={処理中} \
  &statusId[]={レビュー中} \
  &count=100"
```

取得するフィールドで特に重要なのは、以下の3つです。

| フィールド | 用途 |
|---|---|
| `dueDate` | 当初計画の基準日。この日を過ぎていれば「当初計画からの遅延」 |
| `milestone` | 現在のリリース計画（常に最新化されている）。実際のリリース予定 |
| `estimatedHours` | 概算の見積り工数。消化率の算出に使用 |

**`dueDate` と `milestone` の差分**が、スケジュールスリップを検出する鍵です。

```
milestone が dueDate より後  → スリップ発生（リリースが後ろ倒し）
milestone が dueDate と同じ  → 当初計画通り
milestone の期日を過ぎている → 遅延（最も深刻）
```

#### 2. Stock（社内情報共有ツール）— 「昨日何をしたか」の活動データ

メンバーは各自のノート上に、自分用のメモとして日次の作業内容を記録しています。これはもともと報告用ではなく、**個人の作業ログ**です。

AIエージェントは、DB から直接このメッセージを読み取ります（読取専用のコンテナ経由）。

```ruby
# 特定メンバーの当日メッセージを取得（イメージ）
stock = Stock.where(group_id: member_group_id)
             .order(created_at: :desc).first
chats = stock.chat_group.chats
             .where("created_at >= ?", Date.today)
             .order(created_at: :asc)
```

ここで重要なのは、**記事本文（ストック本体）は取得しない**ことです。メッセージ（チャット）のみを取得します。これにより、個人的なメモの詳細まで覗くことなく、「昨日の作業時間と作業内容」だけを把握できます。

#### 3. GitHub（コード管理）— 「コードが正」の客観データ

> 「コードが正」— 口頭の進捗報告よりも、PRの状態が真実を語る。

GitHubからは、PRの状態を進捗ステージとして分類します。

```bash
# PR の詳細情報を取得
gh pr list --repo {org}/{repo} --state all --limit 25 \
  --json number,title,author,state,baseRefName,headRefName, \
         reviewDecision,updatedAt,mergedAt,commits,reviews
```

取得したデータから、以下の **7段階の進捗ステージ** に自動分類します。

| ステージ | 判定条件 | 意味 |
|---|---|---|
| ✅ マージ済み | `state=MERGED` | 実装・レビュー完了 |
| 🟢 承認済み | `reviewDecision=APPROVED` & Open | レビュー通過、マージ待ち |
| 🔵 レビュー中 | レビューコメントあり & Open | フィードバック対応中 |
| 🟡 レビュー待ち | Open & レビュアー未割当 | 実装完了だがレビュー未着手 |
| 🔧 開発中 | Open & 7日以内に更新あり | 活発にコミットしている |
| ⏸️ 停滞 | Open & 7日以上更新なし | 何らかの理由で中断 |
| — PR未作成 | Backlog「処理中」だがPRなし | ローカル開発中 or 未着手 |

このステージ分類によって、**「Backlog上は処理中だが、PRが存在しない」「APPROVEDなのにマージされていない」**といった乖離を自動検出できます。

### レポートの構成

AIエージェントが出力するレポートは、5つのセクションで構成されています。

```markdown
# デイリーレポート（YYYY-MM-DD HH:MM時点）

## 1. チーム全体サマリ（1テーブルで俯瞰）
## 2. 進捗アラート（要対応の課題を検出）
## 3. メンバー別詳細（Backlog × Stock × GitHub の突合）
## 4. 直近更新された課題（過去24時間）
## 5. PO判断事項（担当未割当・優先度調整の提案）
```

特にサマリテーブルでは、**各タスクに進捗ステージを付与**して表示します。

```markdown
| メンバー | 進行中 | MS超過 | 進捗判定 | 主な作業状況 |
|---|---|---|---|---|
| Aさん | 9件 | 1件 | 🔴 遅延 | 機能X 🔧開発中 / 機能Y 🟡レビュー待ち |
| Bさん | 8件 | 0件 | 🟡 注意 | 機能Z PR未作成 / 機能W 🟢承認済み |
```

こうすることで、PMは**テーブルを一目見るだけ**で、チーム全体の状況と「いまどこがボトルネックか」を把握できます。

## 結果

### 定量的な効果

| 指標 | Before | After |
|---|---|---|
| PM の毎朝の状況確認時間 | 30〜45分（手動巡回） | 5分（レポート確認） |
| メンバーの報告作業 | 不定（0〜15分/人） | **0分**（追加作業なし） |
| 進捗状況の把握精度 | 主観的・断片的 | 3データソース突合で客観的 |
| ボトルネック検出 | 週次定例で発覚 | **デイリーで自動検出** |

### 定性的な効果

**1. 「PRが止まっている」を即座に検出**

レポートを導入して最初に発見したのは、**12本のOpen PRがレビュー未着手で滞留**していたことです。あるメンバーは6本のPRを出していましたが、そのうち5本が1〜3週間更新されていませんでした。

従来であれば、週次の定例ミーティングで「なんか止まってるPRない？」と聞くまで気づかなかったものが、レポートの進捗ステージ分析（⏸️停滞の検出）によって初日から可視化されました。

**2. 「Backlog上は処理中だが、コードが存在しない」の検出**

ある課題がBacklog上で「処理中」ステータスになっていたものの、GitHubに対応するPRもブランチも存在しなかったケースがありました。これは**未着手か、ローカルで作業中のまま長期間pushされていない**ことを示唆します。

従来の「Backlogのステータスだけを見る」運用では気づけなかった乖離です。

**3. 「APPROVED済みなのに未マージ」の発見**

コードレビューが完了し、APPROVEDになっているにもかかわらずマージされていないPRが複数見つかりました。リリースタイミングの調整なのか、依存関係の問題なのか — いずれにせよ、PMが気づいて確認すべき事項でした。

### 進捗判定のルール設計で得た学び

最初は `dueDate`（当初の期限日）を超過したら即「遅延」と判定していましたが、実運用で以下のことがわかりました。

- `dueDate` は当初計画であり、**変更されないことが多い**
- `milestone` は常に最新化されており、**実際のリリース計画を反映**している
- `milestone` が `dueDate` より後ろにズレているのは「スリップ」だが、**週次定例で関係者と合意済み**であることが多い

そこで、判定ロジックを以下のように調整しました。

```
🔴 遅延（要対応）：milestone の期日を超過している
🟡 注意（参考情報）：milestone > dueDate だが、milestone 期日内
🟢 オンスケ：milestone 期日内
```

**「スリップ = 即遅延」ではない**。この判断をルール化したことで、レポートのノイズが減り、本当に対応が必要な課題だけがアラートとして浮かび上がるようになりました。

## 技術的なポイント

### AGENTS.md による「レポート仕様のコード化」

この仕組みで最も重要だったのは、**レポートの仕様自体をコードとして管理した**ことです。

具体的には、リポジトリの `AGENTS.md` に以下を全て記述しています。

- データ収集の手順（APIエンドポイント、クエリパラメータ）
- 進捗ステージの判定ロジック（7段階の条件定義）
- レポートのテンプレート（セクション構成、テーブル形式）
- 判定ルール（スリップの扱い、PO行の表示方法など）

これにより、AIエージェントが変わっても、セッションが変わっても、**同じ品質のレポートが再現性を持って生成される**ようになりました。仕様は Git で管理されているので、判定ロジックの変更も差分として追跡できます。

### 「読取専用」の徹底

データ収集は全て**読み取り専用**です。

- Backlog API: GET リクエストのみ
- Stock DB: 読取専用コンテナ経由
- GitHub: `gh pr list` / `gh api` (GET) のみ

AIエージェントが誤ってデータを更新・削除するリスクをアーキテクチャレベルで排除しています。

## 今後の展望

- **トレンド分析**: 週次・月次でのスリップ率、PRリードタイム等の推移を追跡
- **予測的アラート**: 「このペースだと次のマイルストーンに間に合わない」の早期警告
- **メンバーへのフィードバック**: レポートの一部をメンバーにも共有し、セルフマネジメントに活用

## まとめ

- **メンバーに「報告」という追加作業を課さない**のが最も重要な設計判断
- 既存ツール（Backlog / Stock / GitHub）に蓄積されたデータを**AIエージェントが自動で収集・分析**
- PRの状態を7段階のステージに分類し、**「コードが正」の原則で客観的な進捗を把握**
- 判定ロジック（スリップ ≠ 遅延など）は**AGENTS.md にコード化**し、再現性と改善可能性を確保
- 導入初日から「レビュー滞留12本」「APPROVED未マージ2本」を検出 — **手動巡回では見落としていた課題を発見**

「報告のための報告」をなくし、**エンジニアが本来やるべきことに集中できる環境**を作る。AIエージェントの活用は、コード生成だけでなく、こうした開発プロセスの改善にも大きな可能性があると感じています。
