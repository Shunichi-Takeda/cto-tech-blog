# AIエージェントが「見えない不具合」をどう見つけたか — PDFの透明テキストレイヤーをデバッグ可視化で解決した話

PDFの検索機能を使ったとき、ハイライトが実際のテキストと全く異なる場所に表示される——。見た目は完全に正常なのに、検索結果だけがおかしい。そんな「見えない不具合」に遭遇したことはないでしょうか。

この記事では、OCR処理で埋め込まれる**透明テキストレイヤー**のずれに対して、AIエージェントがテキスト座標の修正を試みるも根本解決に至らず、人間が「可視化」を指示したことでバグの本質が明らかになった——という、人間とAIの協働によるデバッグ事例を紹介します。

## はじめに — 「検索結果がずれている」

あるPDFファイルに対してテキスト検索を行ったところ、検索でヒットした箇所のハイライトが、実際のテキストとは全く異なる位置に表示されるという報告がありました。

奇妙なのは、**PDFの見た目は完全に正常**だということです。文字は正しく表示され、レイアウトも崩れていません。にもかかわらず、検索結果だけがずれる。

この症状が発生していたのは、**横向きページ**（PDF の `/Rotate=90` 属性を持つページ）だけでした。

## PDFの透明テキストレイヤーとは

この問題を理解するには、まずスキャンPDFにおける**透明テキストレイヤー**の仕組みを知る必要があります。

スキャンされたPDFは、本質的には**画像**です。文字のように見えていても、PDF内部ではビットマップ画像として格納されています。そのままでは検索もコピーもできません。

これを解決するのがOCR（光学文字認識）処理です。OCRエンジンが画像中の文字を認識し、その位置情報とともに**透明なテキストレイヤー**としてPDFに重ねます。

```
┌─────────────────────┐
│  透明テキストレイヤー   │  ← OCRで生成（目に見えない）
│  "株式会社" at (x,y)  │     検索・コピーに使われる
├─────────────────────┤
│                     │
│   スキャン画像        │  ← ユーザーに見える部分
│   (ビットマップ)      │
│                     │
└─────────────────────┘
```

ユーザーが「検索」すると、PDFビューアはこの透明テキストレイヤーを検索し、該当箇所をハイライト表示します。つまり、**テキストレイヤーの位置がずれていると、ハイライトが実際の文字と異なる場所に表示される**のです。

透明であるがゆえに、見た目には問題がなく、検索して初めてずれに気づく——「見えない不具合」の正体です。

## AIエージェントの初動 — テキスト座標だけを見ていた落とし穴

この問題の調査をAIエージェントに依頼しました。エージェントはまず、OCR結果のテキスト座標とPDFページの寸法を比較する分析を行い、**座標のスケーリング（拡大縮小）がずれている**という仮説を立てました。

実際にスケール補正を施したところ、座標の「位置」は改善されたように見えました。しかし、検索ハイライトのずれは根本的には解消されませんでした。

ここで問題だったのは、エージェントが**テキストの座標（数値）だけを見ていた**ことです。透明テキストレイヤーは目に見えないため、テキストの**向き**や**サイズ**が正しいかどうかを数値だけで判断しようとしていました。結果として、テキストが90°回転して縦書きになっているという根本的な問題を見逃してしまい、座標調整を繰り返す遠回りをしてしまいました。

## 転機 — 「見えないなら、見えるようにしよう」

エージェントが座標調整を重ねても解決しない状況を見て、筆者はエージェントに**テキストレイヤーを可視化する**よう指示しました。透明なものを半透明にして、実際にどう配置されているかを目で見よう、という発想です。

### 可視化の手法

HexaPDF（Ruby製のPDFライブラリ）でテキストレイヤーを追加する際の透明度設定は以下のようになっています。

```ruby
# HexaPDF 0.35.1 / Ruby 3.2
canvas.save_graphics_state do
  canvas.opacity(fill_alpha: 0.0, stroke_alpha: 0.0)  # 完全に透明
  canvas.font(font, size: font_size)
  canvas.text(text, at: [x, y])
end
```

この `fill_alpha: 0.0`（完全透明）を `fill_alpha: 0.5`（半透明）に変更し、さらにテキスト色を赤に設定するだけで、テキストレイヤーの位置が目に見えるようになります。

```ruby
# デバッグ用: 半透明赤色でテキストレイヤーを可視化
canvas.save_graphics_state do
  canvas.opacity(fill_alpha: 0.5, stroke_alpha: 0.8)  # 半透明
  canvas.fill_color(255, 0, 0)                        # 赤色
  canvas.font(font, size: font_size)
  canvas.text(text, at: [x, y])
end
```

この変更を加えてPDFをレンダリングしたところ、エージェントが数値では気づけなかった問題が**一目瞭然**になりました。

### 可視化で初めて分かったこと

**Rotation=0°（標準縦向き）のページ**では、赤色のテキストレイヤーが元のテキスト位置に正確に重なっていました。正常に動作しています。

しかし**Rotation=90°（横向き）のページ**では、テキストレイヤーが**縦書き**で表示され、テキストのサイズも全く異なっていました。

これはテキスト座標の数値だけでは気づけない問題です。座標が「ある程度正しい位置」にあったとしても、テキストの**向き**が90°ずれ、**サイズ**も異なっていれば、検索ハイライトは正しく表示されません。エージェントが座標調整を繰り返しても解決しなかった理由が、可視化によって明確になりました。

## 原因分析 — PDF座標系の罠

### 2つの座標系

PDFの `/Rotate` 属性は、**表示時にページを時計回りに回転する角度**を指定するものです。重要なのは、**内部の座標系（MediaBox座標系）は回転しない**という点です。

```
Rotation=90° のPDF:

【内部（MediaBox座標系）】        【表示（ディスプレイ座標系）】
┌───────┐                      ┌──────────────┐
│       │                      │              │
│       │ 595pt               │              │ 595pt
│       │                      │              │
│       │                      └──────────────┘
└───────┘                           842pt
  842pt

原点は左下、座標系は縦長          見た目は横長に表示される
```

### ミスマッチの発生メカニズム

OCRエンジン（Cloud Vision API）は、PDFを**画像として認識**します。Rotation=90°のPDFを画像化すると、横長の画像として認識されます。つまり、OCRが返すテキスト座標は**表示座標系**（ディスプレイ座標系）に基づいています。

一方、HexaPDF の `canvas.text(at: [x, y])` は**MediaBox座標系**で描画します。

| 座標系 | 原点 | 方向 | 使用箇所 |
|---|---|---|---|
| MediaBox座標系 | 左下 | 回転なし | HexaPDF の canvas |
| 表示座標系 | 左上 | Rotate適用済み | Cloud Vision API |

Rotation=0°では両者が一致するため問題になりませんが、Rotation≠0°ではこのミスマッチにより、テキストの位置と向きが狂います。

## 修正 — CTM（座標変換行列）の導出

### canvas.rotate は使えない

最初に試したのは `canvas.rotate(90)` による回転でした。しかしこれは**回転のみ**で、座標の原点移動を行いません。結果として、テキストの向きは合っても位置がずれたままでした。

### canvas.transform で解決

正しいアプローチは、`canvas.transform` に**座標変換行列（CTM: Current Transformation Matrix）**を渡すことです。CTMは6つのパラメータ `[a, b, c, d, e, f]` を持ち、回転・スケーリング・平行移動を一度に表現できます。

各Rotation値に対するCTMを導出しました。

```ruby
# HexaPDF 0.35.1 / Ruby 3.2
rotation = original_page[:Rotate] || 0
if rotation != 0
  media_box = original_page.box(:media)
  mb_w = media_box.width.to_f   # MediaBoxの幅
  mb_h = media_box.height.to_f  # MediaBoxの高さ
  cv_w = ocr_page.width.to_f    # Cloud Visionが認識した幅
  cv_h = ocr_page.height.to_f   # Cloud Visionが認識した高さ

  case rotation
  when 90
    sx = mb_h / cv_w;  sy = mb_w / cv_h
    canvas.transform(0, sx, -sy, 0, mb_w, 0)
  when 180
    sx = mb_w / cv_w;  sy = mb_h / cv_h
    canvas.transform(-sx, 0, 0, -sy, mb_w, mb_h)
  when 270
    sx = mb_h / cv_w;  sy = mb_w / cv_h
    canvas.transform(0, -sx, sy, 0, 0, mb_h)
  end
end
```

| Rotation | CTM `[a, b, c, d, e, f]` | 意味 |
|---|---|---|
| 0° | Identity（変換不要） | 座標系が一致 |
| 90° | `[0, sx, -sy, 0, mb_w, 0]` | 90°回転＋原点移動 |
| 180° | `[-sx, 0, 0, -sy, mb_w, mb_h]` | 180°回転＋原点移動 |
| 270° | `[0, -sx, sy, 0, 0, mb_h]` | 270°回転＋原点移動 |

修正量はわずか**22行の追加**。既存コード（Rotation=0°のパス）には一切変更を加えていないため、リグレッションリスクも最小限です。

## 検証 — サンプルPDFで全パターンテスト

修正の検証にあたり、AIエージェントは以下のステップを自律的に実行しました。

### 1. テスト用サンプルPDFの作成

PyMuPDFを使い、4ページ構成のPDFを生成しました。各ページに異なるRotation値（0°/90°/180°/270°）を設定し、四隅と中央にマーカーテキストを配置しています。

```python
# PyMuPDF (fitz) 1.23
import fitz

doc = fitz.open()
for rotation in [0, 90, 180, 270]:
    page = doc.new_page(width=595.28, height=841.89)
    page.set_rotation(rotation)
    # 四隅 + 中央にマーカーテキストを配置
    page.insert_text((50, 50), f"左上マーカー", fontname="japan", fontsize=14)
    page.insert_text((width/2, height/2), f"中央マーカー R={rotation}°")
    # ...
doc.save("sample_rotation_test.pdf")
```

### 2. 修正前/修正後の比較

サンプルPDFに対して、修正前（CTMなし）と修正後（CTM適用）の2パターンでテキストレイヤーを追加し、半透明赤色で可視化しました。

| Rotation | 修正前（バグ状態） | 修正後（CTM適用） | 判定 |
|---|---|---|---|
| **0°** | テキスト位置一致 | テキスト位置一致（変化なし） | ✅ リグレッションなし |
| **90°** | テキストが縦書き、位置ずれ | 横書き・正しい位置に配置 | ✅ 修正成功 |
| **180°** | テキストが逆さま、位置ずれ | 正しい向き・正しい位置に配置 | ✅ 修正成功 |
| **270°** | テキストが縦書き（逆）、位置ずれ | 横書き・正しい位置に配置 | ✅ 修正成功 |

### 3. 本番データでの追加検証

サンプルPDFでの検証に加え、本番環境のOCR処理済みPDF約47,000件からランダムに10件を抽出して追加検証を行いました。

10件中2件（20%）がRotation付きのPDFでした。これらに対して修正ロジックを適用した結果、修正前に確認されたマーカーの縦書き・位置ずれが修正後には解消されていることを確認しました。Rotation=0°のファイル（21ページの大きなPDFを含む）でのリグレッションもありませんでした。

## 学び — 人間とAIの協働デバッグ

### AIが陥る「数値だけで判断する罠」

今回の事例でAIエージェントは、テキストの座標（数値）だけを見て問題を解こうとしました。座標が「だいたい合っている」ように見えると、そこで満足してしまい、テキストの向きやサイズという**視覚的な情報**を見落としていたのです。

これは、AIエージェントが「見えないもの」を扱う際の典型的な落とし穴と言えます。数値やログだけでは捉えきれない問題が存在し、それは**人間の「見てみよう」という直感**によって初めて露わになることがあります。

### 「可視化してみて」という一言の威力

筆者がエージェントに伝えたのは「テキストレイヤーを可視化してみて」というシンプルな指示でした。しかしこの一言が、エージェントが座標調整を繰り返す袋小路から脱出し、根本原因に到達するきっかけになりました。

可視化を指示された後のエージェントは、原因分析→CTM導出→サンプルPDF作成→全パターン検証までを高速に実行しました。**方向性を示すのは人間、実行力を発揮するのはAI**——この役割分担が効果的に機能した事例です。

### 数学的問題を一気通貫で解く

方向性が定まった後のAIエージェントの実行力は目を見張るものがありました。座標変換行列の導出は、PDF仕様の理解、線形代数の知識、そしてライブラリのAPI仕様を横断的に把握する必要がある作業です。エージェントはこれらを統合し、4パターンのCTMを導出・検証コード作成・実行・結果確認までを一連の流れで実行しました。

### 予防的な網羅対応

実際に報告されたのは Rotation=90° の1パターンだけでしたが、AIエージェントは180°/270°のパターンも同時に洗い出し、修正と検証を行いました。追加のコードはわずか6行ですが、将来的なバグ報告を未然に防ぐことができます。

## まとめ

- **透明テキストレイヤーのバグ**は、見た目が正常なため通常のテストでは検出不能
- **AIエージェントは座標（数値）だけを見て**向きの問題を見逃し、根本解決が遠のいた
- **人間の「可視化してみて」という指示**が、袋小路から抜け出す転機になった
- 可視化後は、**原因特定→CTM導出→全パターン検証**をAIが一気通貫で実行
- **CTM（座標変換行列）**により、22行の追加で全Rotation値に対応可能

AIエージェントを開発ワークフローに組み込む際、「AIに任せきりにしない」ことの重要性を実感した事例です。数値の世界に閉じたAIに**「目で見てみよう」と促す**のは、今のところ人間の役割です。一方で、方向性が定まった後の実行速度はAIの圧倒的な強みです。**人間が方向を示し、AIが全力で走る**——この協働モデルが、「見えない不具合」を解決する鍵でした。
