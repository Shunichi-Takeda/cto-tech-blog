# AGENTS.md — CTO Tech Blog

本リポジトリは、AIエージェントと連携して技術記事の草案を管理するリポジトリである。
普段は動作しないが、業務中に技術的な発信・共有に値する内容を検出した場合、指示に基づき記事草案を作成・コミットする。

---

## 1. リポジトリの目的

| 目的 | 詳細 |
|---|---|
| **メディア発信力の向上** | Stock社としての技術的プレゼンスを確立する |
| **採用力の向上** | エンジニア候補に対して技術力・開発文化を可視化する |
| **エンジニアの発信の場** | チームメンバーが技術的知見を社外に発信する基盤を提供する |

## 2. ディレクトリ構成

```
cto-tech-blog/
├── AGENTS.md              # 本ファイル（AIエージェント向け運用ガイド）
├── README.md              # リポジトリ概要
├── articles/
│   ├── ideas/             # 記事アイデア（ネタ出し段階）
│   │   └── YYYY-MM-DD_slug.md
│   ├── drafts/            # 草案（執筆中・レビュー前）
│   │   └── YYYY-MM-DD_slug.md
│   └── published/         # 公開済み記事（公開後のアーカイブ）
│       └── YYYY-MM-DD_slug.md
└── templates/
    ├── article_template.md    # 記事テンプレート
    └── idea_template.md       # アイデアテンプレート
```

### ステージフロー

```
ideas/ → drafts/ → published/
 ネタ出し   草案作成    公開済み
```

- **ideas/**: 業務中に検出した「記事にできそうなネタ」をメモレベルで蓄積
- **drafts/**: 記事として構成・執筆した草案。CTO レビュー後に公開判断
- **published/**: 公開済み記事のアーカイブ（公開先URLを frontmatter に記録）

## 3. ファイル命名規則

```
YYYY-MM-DD_slug.md
```

| 要素 | 説明 | 例 |
|---|---|---|
| `YYYY-MM-DD` | 記事作成日（アイデア起案日 or 草案着手日） | `2026-06-23` |
| `slug` | 記事を一意に識別する英数字スラッグ | `ai-agent-daily-report-automation` |

例: `2026-06-23_ai-agent-daily-report-automation.md`

## 4. 記事 Frontmatter 仕様

各記事ファイルの先頭に YAML frontmatter を配置する。

```yaml
---
title: "記事タイトル"
slug: "url-friendly-slug"
author: "著者名"
author_github: "github_username"
date: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
stage: "idea | draft | review | published"
tags:
  - タグ1
  - タグ2
category: "カテゴリ"
description: "記事の概要（1-2文）"
source_context: "この記事のきっかけとなった業務・技術的背景"
published_url: ""  # 公開後に記入
---
```

### stage の遷移

| stage | 意味 | 場所 |
|---|---|---|
| `idea` | ネタ・アイデア段階 | `articles/ideas/` |
| `draft` | 草案として執筆中 | `articles/drafts/` |
| `review` | CTOレビュー中 | `articles/drafts/`（stage のみ変更） |
| `published` | 公開済み | `articles/published/` |

## 5. AIエージェントの運用ルール

### 5.1 記事ネタの検出基準

業務中に以下のいずれかに該当する技術的要素を検出した場合、記事ネタとして `articles/ideas/` に起票する。

| カテゴリ | 判定基準 | 例 |
|---|---|---|
| **技術的課題解決** | 非自明な方法で技術的課題を解決した | N+1問題の体系的解決、パフォーマンスチューニング |
| **アーキテクチャ判断** | 設計上の重要な意思決定を行った | マイクロサービス分割、DB設計の方針転換 |
| **ツール・プロセス改善** | 開発プロセスやツールを改善した | CI/CD パイプラインの最適化、AIエージェント活用 |
| **障害対応・ポストモーテム** | 本番障害の原因分析と対策が完了した | メモリリーク調査、データ不整合の修復 |
| **新技術導入** | 新しい技術・ライブラリを本番導入した | OTel導入、新しいAWS サービスの活用 |
| **チーム・組織** | エンジニア組織の運用に関する知見 | コードレビュー文化、リモートワーク運用 |
| **AIエージェント活用** | AIエージェントの業務活用事例 | デイリーレポート自動生成、設計レビュー自動化 |

### 5.2 記事作成フロー

```
1. ネタ検出
   └─ AIエージェントが業務中に検出 or ユーザーが指示
   └─ articles/ideas/ にアイデアファイルを作成

2. 草案作成（ユーザーの指示があった場合のみ）
   └─ ideas/ のファイルを drafts/ に昇格
   └─ templates/article_template.md に基づき本文を構成
   └─ stage を "draft" に更新

3. レビュー
   └─ ユーザー（CTO）がレビュー・フィードバック
   └─ stage を "review" に更新

4. 公開
   └─ 公開先プラットフォームへの投稿
   └─ published/ にファイルを移動
   └─ published_url を記入
   └─ stage を "published" に更新
```

### 5.3 記事の品質基準

#### 必須要件

- [ ] タイトルが具体的で、読者が得られる知見が明確である
- [ ] 「課題 → アプローチ → 結果 → 学び」の構造になっている
- [ ] 技術的な正確性が担保されている（コード例があれば動作確認済み）
- [ ] Stock社固有の機密情報（顧客名・内部数値・非公開仕様）を含まない
- [ ] 読者（外部エンジニア）にとって再現可能な形で記述されている

#### 推奨事項

- [ ] コード例やコマンド例を含む（抽象論だけでなく具体性がある）
- [ ] 図やダイアグラムで視覚的に説明している
- [ ] 類似の課題に対する既存のアプローチとの比較がある
- [ ] 「なぜこの方法を選んだか」の意思決定プロセスが記述されている

### 5.4 機密情報ガード

記事作成時に以下の情報が含まれていないことを**必ず確認する**。

| NG情報 | 例 |
|---|---|
| **顧客固有情報** | 顧客名、契約内容、問い合わせ内容 |
| **内部数値** | 売上、MRR、ユーザー数、具体的なサーバー台数 |
| **非公開仕様** | セキュリティ実装の詳細、認証フローの具体的実装 |
| **APIキー・認証情報** | 環境変数名は可、値は不可 |
| **社内URL** | Backlog、Stock、Slack等の内部URL |
| **個人情報** | 社員のフルネーム（GitHub アカウント名は可） |

> ⚠️ 判断がつかない場合は**含めない**。CTO レビュー時に個別判断する。

### 5.5 記事のトーン・スタイル

| 項目 | ガイドライン |
|---|---|
| **文体** | です・ます調。技術的に正確かつ親しみやすい文体 |
| **読者想定** | 中〜上級のWebエンジニア |
| **長さ** | 2,000〜5,000文字（目安） |
| **構成** | 見出し（h2/h3）を適切に使い、スキャンしやすくする |
| **コード例** | 実際に動くコードを掲載。言語・フレームワークのバージョンを明記 |
| **CTO視点の記事** | 技術選定の背景・組織的な判断理由を重視する |

## 7. AIエージェント間連携

本リポジトリは他リポジトリの AGENTS.md と以下のように連携する。

```
ai-knowledge (横断ナレッジ)
  └─ 業務中に検出した技術的知見
       ↓ ネタ検出・記事起票
cto-tech-blog (本リポジトリ)
  └─ articles/ideas/ → drafts/ → published/

narekan-project-management (デイリーレポート)
  └─ 開発活動の中で記事化すべき要素を検出
       ↓ ネタ検出・記事起票
cto-tech-blog (本リポジトリ)
```

### 連携時のルール

1. **他リポジトリのAIエージェントは直接 cto-tech-blog にコミットしない** — ネタの起票は指示ベースで行う
2. **記事作成はCTOの指示があった場合のみ** — 自動で草案を量産しない
3. **ネタの提案は積極的に行う** — 「これは記事にできそうです」という提案は歓迎

## 8. 重要ルール

1. **記事の作成・草案のコミットはユーザー（CTO）の指示があった場合のみ実行する**
2. **機密情報ガード（§5.4）は絶対に遵守する**
3. **ネタの提案は積極的に行うが、勝手に草案を作成しない**
4. **公開判断は必ずCTOが行う** — AIエージェントが公開可否を判断しない
5. **他リポジトリの機密コードを記事にそのまま転載しない** — 汎化・抽象化した上で掲載する
